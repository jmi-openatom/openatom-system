# 兄弟社团展示页面需求文档

## 1. 文档信息

| 项目名称 | JMI-OPENATOM 兄弟社团展示页 | 版本号 | V1.1.0 |
| --- | --- | --- | --- |
| 所属系统 | OpenAtom System 官网 | 页面类型 | 公共展示页面 |
| 建议路由 | `/partners` | 导航名称 | 兄弟社团 |
| 目标终端 | PC、平板、手机 | 文档状态 | 待开发 |

## 2. 项目背景

JMI-OPENATOM 官网目前主要展示本社团的活动、文章、成员与成果，缺少对校内外友好社团和合作组织的集中展示入口。新增“兄弟社团”页面，用于展示合作社团的品牌 Logo、基本介绍和官方链接，帮助访问者了解相关组织，并建立清晰、可信的社团协作网络。

## 3. 产品目标

- 集中展示与 JMI-OPENATOM 保持交流或合作关系的社团。
- 让访问者快速识别社团 Logo、名称、定位和主要方向。
- 提供明确、安全的外部访问入口。
- 在首页底部展示精选兄弟社团，让访客无需进入独立页面也能快速发现合作组织。
- 保持与现有官网一致的视觉语言、深浅色主题和响应式体验。
- 为后续后台维护、排序、上下架和扩展联系方式预留数据结构。

## 4. 功能范围

### 4.1 本期范围

- 新增公开页面“兄弟社团”。
- 在首页最后一个内容区与公共页脚之间新增“兄弟社团”精选展示区。
- 在官网导航和移动端抽屉导航中增加入口。
- 展示社团 Logo、社团名称、简介和官方链接。
- 支持点击卡片或按钮打开社团官方页面。
- 支持加载态、空状态、图片加载失败状态。
- 支持 PC、平板和手机响应式布局。
- 支持浅色模式和深色模式。

### 4.2 后续扩展

- 后台新增兄弟社团管理功能。
- 支持合作类型、所属学校、地区、技术方向等标签。
- 支持置顶、推荐、排序和定时发布。
- 支持展示微信公众号、QQ 群、GitHub、Gitee 等多个链接。
- 支持记录外链点击量。

### 4.3 暂不纳入本期

- 在线申请成为兄弟社团。
- 社团之间的即时聊天或站内消息。
- 对外部社团网站内容进行抓取或同步。
- 对兄弟社团进行评分、排名或评论。

## 5. 页面信息架构

页面从上到下包含以下区域：

1. 官网公共页头。
2. 页面首屏介绍区。
3. 兄弟社团卡片列表。
4. 空状态或加载失败提示。
5. 官网公共页脚。

### 5.1 首屏介绍区

- 小标题：`开放协作`
- 主标题：`兄弟社团`
- 推荐介绍文案：

  > 开源因连接而生长。这里汇聚与我们并肩交流、共同实践的校园社团与开源组织。

- 可选辅助信息：当前已展示的社团数量。

### 5.2 社团卡片

每张卡片必须包含：

| 内容 | 必填 | 展示规则 |
| --- | --- | --- |
| 社团 Logo | 是 | 保持原始比例，不拉伸；建议使用 SVG、WebP 或透明 PNG |
| 社团名称 | 是 | 最多两行，超出时使用省略号 |
| 社团简介 | 是 | 建议 40～120 个中文字符，最多显示四行 |
| 官方链接 | 是 | 显示“访问官网”或“了解更多”按钮 |

可选展示内容：

- 所属学校或组织。
- 合作类型，例如“校园社团”“开源社区”“技术协会”。
- 技术方向标签，例如“Linux”“鸿蒙”“人工智能”“Web 开发”。

### 5.3 首页底部精选区

首页底部必须展示兄弟社团预览，位置为首页最后一个业务内容区之后、官网公共页脚之前。

推荐内容结构：

1. 小标题：`开放协作`。
2. 主标题：`兄弟社团`。
3. 简介：`与志同道合的校园社团和开源组织并肩交流，共同建设开放、共享的技术社区。`
4. 精选社团卡片列表。
5. 站内入口按钮：`查看全部兄弟社团`，跳转至 `/partners`。

展示规则：

- 仅展示 `status = published` 且 `featured = true` 的社团。
- 最多展示 4 个，按 `sortOrder` 从小到大排列；不足 4 个时展示实际数量。
- 若没有符合条件的数据，则隐藏整个首页精选区，不显示空状态或无效入口。
- 首页卡片使用紧凑变体，保留 Logo、社团名称、两行简介和“访问官网”入口。
- “查看全部兄弟社团”必须是站内链接；“访问官网”必须遵守外链安全规范。

## 6. 交互需求

### 6.1 外链访问

- 点击“访问官网”按钮后，在新标签页打开链接。
- 外链必须使用 `target="_blank"` 和 `rel="noopener noreferrer"`。
- 按钮旁使用统一的外链 SVG 图标，不使用 Emoji。
- 鼠标悬停时显示轻微的边框、背景或位移反馈，动画时长控制在 150～250ms。
- 键盘用户可以通过 Tab 聚焦卡片中的访问按钮，并看到清晰的焦点样式。
- 移动端按钮触控区域不小于 44×44px。

### 6.2 Logo 状态

- Logo 加载期间预留固定尺寸，避免页面布局跳动。
- Logo 加载失败时显示社团名称首字或通用社团占位图。
- 必须提供有意义的 `alt`，格式建议为“{社团名称} Logo”。
- 不允许使用 CSS 强制改变 Logo 宽高比例。

### 6.3 列表状态

- 加载中：显示与实际卡片尺寸一致的骨架屏。
- 无数据：显示“暂未添加兄弟社团”，并保留完整页面结构。
- 请求失败：显示“兄弟社团加载失败”，提供“重新加载”按钮。
- 部分 Logo 失败时不能影响其他社团卡片显示。

### 6.4 首页精选区交互

- 点击“查看全部兄弟社团”进入 `/partners`，不打开新标签页。
- 首页精选卡片中的“访问官网”在新标签页打开，并提供清晰的外链提示。
- 不采用自动轮播，避免内容跳动、触控冲突和可访问性问题。
- 卡片悬停、按下和键盘聚焦反馈不得引发布局位移。

## 7. 视觉与响应式规范

### 7.1 视觉方向

- 沿用官网现有黑、白、灰色设计令牌，不新增独立品牌主色。
- 卡片使用现有页面背景、边框、圆角和文字变量。
- 兄弟社团自身的品牌色仅出现在官方 Logo 中，不用其品牌色改变整张卡片。
- 页面整体保持简洁、可信、内容优先，避免过度装饰。

### 7.2 卡片布局

| 视口宽度 | 推荐列数 | 页面边距 | 卡片间距 |
| --- | --- | --- | --- |
| ≥ 1200px | 3 列 | 24px 以上 | 20～24px |
| 768～1199px | 2 列 | 24px | 16～20px |
| < 768px | 1 列 | 16px | 16px |

卡片建议结构：

```text
┌──────────────────────────────────┐
│  [Logo]  社团名称                │
│          所属学校 / 社团类型      │
│                                  │
│  社团简介，说明主要方向、特色与   │
│  同 JMI-OPENATOM 的协作关系。     │
│                                  │
│  [访问官网 ↗]                    │
└──────────────────────────────────┘
```

### 7.3 Logo 尺寸

- PC：推荐 72×72px 的展示区域。
- 手机：推荐 56×56px 的展示区域。
- 展示区域使用 `object-fit: contain`。
- Logo 周围保留至少 8px 安全留白。

### 7.4 深浅色模式

- 卡片背景、边框、正文和辅助文字使用现有语义变量。
- Logo 不进行反色处理。
- 若 Logo 仅适用于浅色背景，应在 Logo 容器内保留中性浅色底和细边框。
- 正文与背景对比度至少达到 WCAG AA 4.5:1。

### 7.5 首页精选区响应式布局

| 视口宽度 | 首页展示列数 | Logo 展示区域 | 说明 |
| --- | --- | --- | --- |
| ≥ 1200px | 4 列 | 56×56px | 单行展示最多 4 个精选社团 |
| 768～1199px | 2 列 | 56×56px | 自动换为两行，不横向滚动 |
| < 768px | 1 列 | 48×48px | 纵向排列，避免嵌套滑动手势 |

- 首页精选区的最大内容宽度和左右边距应与首页其他区块保持一致。
- 移动端卡片宽度不得超出视口，按钮触控区域不得小于 44×44px。
- 简介在首页卡片中最多显示两行，完整介绍在 `/partners` 页面展示。
- 首页精选区与公共页脚之间保持与现有首页区块一致的垂直间距。

## 8. 数据结构建议

### 8.1 前端类型

```ts
export interface PartnerClub {
  id: number
  name: string
  logoUrl: string
  description: string
  websiteUrl: string
  organization?: string
  category?: string
  tags?: string[]
  sortOrder: number
  featured: boolean
  status: 'draft' | 'published' | 'disabled'
}
```

### 8.2 示例数据

```json
{
  "id": 1,
  "name": "示例开源技术社",
  "logoUrl": "/api/v1/files/images/example-club-logo.png",
  "description": "专注开源技术学习、项目实践和校园技术交流，与 JMI-OPENATOM 共同推动校园开源文化建设。",
  "websiteUrl": "https://example.org",
  "organization": "示例职业技术学院",
  "category": "校园技术社团",
  "tags": ["开源", "Linux", "项目实践"],
  "sortOrder": 10,
  "featured": true,
  "status": "published"
}
```

### 8.3 数据规则

- `name`：必填，建议不超过 50 个字符。
- `description`：必填，建议不超过 300 个字符。
- `logoUrl`：必填，必须是系统文件地址或可信 HTTPS 地址。
- `websiteUrl`：必填，仅允许 `https://` 或经过确认的 `http://` 地址。
- `sortOrder`：数字越小越靠前；相同时按创建时间倒序排列。
- 公开页面仅返回 `status = published` 的数据。

## 9. 接口建议

### 9.1 官网查询接口

```http
GET /api/v1/site/partner-clubs
```

首页推荐查询建议复用同一接口，并增加可选参数：

```http
GET /api/v1/site/partner-clubs?featured=true&limit=4
```

- `featured`：为 `true` 时仅返回推荐社团。
- `limit`：限制返回数量，首页固定传入 `4`。
- 无论是否携带参数，接口都只能返回 `status = published` 的数据。
- 接口返回顺序统一按 `sortOrder` 升序排列。

响应示例：

```json
{
  "code": 200,
  "message": "success",
  "data": [
    {
      "id": 1,
      "name": "示例开源技术社",
      "logoUrl": "/api/v1/files/images/example-club-logo.png",
      "description": "专注开源技术学习、项目实践和校园技术交流。",
      "websiteUrl": "https://example.org",
      "organization": "示例职业技术学院",
      "category": "校园技术社团",
      "tags": ["开源", "Linux"],
      "featured": true
    }
  ]
}
```

### 9.2 后台管理接口（后续）

- `GET /api/v1/admin/partner-clubs`：分页查询。
- `POST /api/v1/admin/partner-clubs`：新增。
- `PUT /api/v1/admin/partner-clubs/{id}`：编辑。
- `DELETE /api/v1/admin/partner-clubs/{id}`：删除。
- `PUT /api/v1/admin/partner-clubs/{id}/status`：上下架。
- `PUT /api/v1/admin/partner-clubs/sort`：批量排序。

## 10. 前端实现建议

- 页面组件建议：`frontend/web_pc/src/views/site/Partners.vue`。
- 路由名称建议：`site-partners`。
- 在 `SiteLayout.vue` 的桌面导航和移动抽屉导航中同时增加“兄弟社团”。
- 社团卡片建议拆分为 `PartnerClubCard.vue`，便于后续在首页复用。
- 首页区块建议拆分为 `HomePartnerClubsSection.vue`，并在首页最后一个业务区块之后引入。
- `PartnerClubCard.vue` 建议提供 `default` 和 `compact` 两种展示变体，独立页面使用 `default`，首页使用 `compact`。
- 首页接口请求使用 `featured=true&limit=4`；无数据时组件返回空内容，不占据页面高度。
- 外链图标使用项目现有 Element Plus 或 Lucide 图标库。
- 图片应声明宽高或 `aspect-ratio`，避免累计布局偏移。
- 卡片列表低于首屏时可对 Logo 使用 `loading="lazy"`。
- 动画应尊重 `prefers-reduced-motion: reduce`。

## 11. 安全要求

- 后端保存链接前校验 URL 协议，拒绝 `javascript:`、`data:` 等危险协议。
- 外部链接不得携带当前系统 Token、用户信息或内部查询参数。
- 富文本简介首期不开放，仅允许纯文本。
- Logo 上传沿用现有图片上传的类型、大小和权限校验。
- 前端渲染简介时不得使用未经清洗的 `v-html`。

## 12. 验收标准

### 12.1 功能验收

- 官网导航可以进入“兄弟社团”页面。
- 页面正确展示所有已发布社团的 Logo、名称、简介和官方链接。
- 点击访问按钮会在新标签页打开正确地址。
- 草稿或已停用社团不会出现在公开页面。
- Logo 加载失败时有可识别的占位内容。
- 无数据和接口失败时有明确提示。
- 首页最下方、公共页脚之前可以看到最多 4 个已发布且已推荐的兄弟社团。
- 首页精选社团的顺序与后台 `sortOrder` 一致。
- 首页“查看全部兄弟社团”可以正确跳转至 `/partners`。
- 没有推荐社团时，首页不显示该区块，也不留下多余间距。

### 12.2 视觉验收

- PC、平板和手机端均无横向滚动条。
- 375px、430px、768px、1024px 和 1440px 视口下布局正常。
- 所有 Logo 保持原始比例且不模糊、不拉伸。
- 同一行卡片高度和内容对齐自然。
- 深色和浅色模式下文字、边框、按钮均清晰可见。
- 交互按钮的触控区域不小于 44×44px。
- 首页精选区在桌面端 4 列、平板端 2 列、手机端 1 列显示，且不出现横向滚动或卡片挤压。
- 首页精选区与现有首页、公共页脚的视觉风格和内容宽度保持一致。

### 12.3 无障碍验收

- 页面标题层级从 `h1` 到 `h2` 顺序正确。
- Logo 具有正确的替代文本。
- 外链按钮具有可理解的无障碍名称，例如“访问示例开源技术社官网”。
- 键盘可以访问所有外链按钮，焦点样式清晰。
- 颜色不是表达交互状态的唯一方式。

## 13. 建议实施顺序

1. 确认页面名称、路由和首批兄弟社团资料。
2. 建立 `PartnerClub` 数据表和查询接口，或先以静态配置完成首期展示。
3. 开发可复用卡片组件、独立页面和响应式样式。
4. 开发首页底部精选区，并接入推荐数据和“查看全部”入口。
5. 接入导航、深浅色模式、加载和错误状态。
6. 完成外链安全、键盘操作和多尺寸验收。
7. 后续补充后台管理、排序和上下架能力。

## 14. 上线资料清单

每个兄弟社团上线前需要提供：

- 正式社团名称。
- 官方 Logo 原文件，优先 SVG，其次透明背景 PNG/WebP。
- 40～120 字社团简介。
- 官方网站或官方主页链接。
- 所属学校或组织（可选）。
- 社团类型和技术方向标签（可选）。
- 是否推荐展示及期望排序（可选）。
