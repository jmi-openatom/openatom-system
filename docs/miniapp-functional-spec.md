## 2. 项目背景

OpenAtom System 已具备社团门户、活动管理、招新申请、通知、动态表单、签到及用户体系。PC 端适合信息维护和管理，小程序则承担学生和社团成员在微信内的高频轻量操作。

本期目标不是建设管理后台，而是交付一个可以上线使用的用户端入口：用户能了解社团、浏览和报名活动、提交招新申请、查看进度与消息，并在线下活动现场扫码签到。

## 3. 产品定位与目标

### 3.1 一句话定位

面向高校学生和社团成员的 OpenAtom 移动服务入口。

### 3.2 产品目标

1. 降低用户了解社团、参加活动和提交入会申请的操作成本。
2. 打通活动发布、报名、签到和通知触达的完整闭环。
3. 复用现有 OpenAtom 后端账号与业务数据，避免形成独立数据孤岛。
4. 支持后台生成小程序路径，将用户直接带到指定招新或动态表单页面。

### 3.3 首版成功指标

| 指标 | 建议目标 | 统计口径 |
| --- | --- | --- |
| 首页成功加载率 | ≥ 99% | 首页接口正常返回且核心内容完成渲染 |
| 微信登录成功率 | ≥ 95% | 获取微信 code 后成功换取平台登录态 |
| 活动报名成功率 | ≥ 98% | 点击报名并通过前端校验后的成功请求占比 |
| 招新申请提交成功率 | ≥ 98% | 通过表单校验后的成功请求占比 |
| 扫码签到成功率 | ≥ 98% | 扫描有效且未过期签到码后的成功请求占比 |
| 核心页面崩溃率 | < 0.5% | 有效访问中发生脚本异常或白屏的比例 |

## 4. 用户角色与权限

| 角色 | 身份说明 | 可用功能 |
| --- | --- | --- |
| 游客 | 未登录用户 | 浏览首页、活动、招新；查看详情；按后台规则提交公开招新或动态表单 |
| 登录用户 | 已通过账号密码或微信登录 | 游客全部能力；活动报名、消息、申请进度、扫码签到、微信绑定 |
| 社团成员 | 登录用户且存在有效成员关系 | 登录用户全部能力；接收成员通知；参与内部活动与签到 |
| 管理员 | 后台管理角色 | 本期小程序不提供管理功能，统一使用 PC 管理后台 |

权限原则：

- 公开内容允许游客查看，涉及个人数据和内部操作必须登录。
- 前端的按钮隐藏或禁用只用于体验控制，最终权限必须由后端校验。
- 访问受限页面时，记录原目标页面；登录成功后优先回到原页面。
- 登录过期时清理本地会话，引导重新登录，不允许出现无限重试。

## 7. 核心用户流程

### 7.1 微信登录

1. 用户点击“微信快捷登录”。
2. 小程序调用 `uni.login` 获取一次性 code。
3. 客户端提交 code 至 `POST /auth/miniapp-login`。
4. 后端向微信服务端换取 openid，并查询已绑定的平台用户。
5. 已绑定时返回 Token、用户、角色和权限；客户端保存会话。
6. 未绑定时进入“登录已有账号并绑定”或“按平台规则注册”流程。
7. 登录成功后返回原业务页面；无原页面时进入首页或个人中心。

业务规则：

- 微信 code 只能使用一次，不得缓存或写入日志。
- 小程序端不得保存 AppSecret，所有微信服务端交换均由后端完成。
- Token 只存储在小程序本地安全存储中，请求统一携带。
- 用户主动退出时同时调用后端退出接口并清理本地 Token、用户和权限缓存。

### 7.2 活动报名

1. 用户从首页或活动列表进入活动详情。
2. 页面展示活动时间、地点、状态、介绍、报名时间和是否需要报名。
3. 不需要报名时显示“无需报名”；需要报名时检查登录态和报名窗口。
4. 未登录用户点击报名，进入登录页并记录活动详情地址。
5. 已登录用户确认报名，客户端发起报名请求。
6. 后端校验活动状态、报名时间、名额、重复报名和用户资格。
7. 成功后更新按钮为“已报名”，并提示用户留意活动通知。

状态规则：

| 条件 | 按钮文案 | 是否可操作 |
| --- | --- | --- |
| 不要求报名 | 无需报名 | 否 |
| 报名未开始 | 报名未开始 | 否 |
| 报名进行中且未报名 | 立即报名 | 是 |
| 已报名 | 已报名 | 否 |
| 名额已满 | 名额已满 | 否 |
| 报名已截止 | 报名已截止 | 否 |
| 活动已结束或取消 | 活动已结束/已取消 | 否 |

### 7.3 招新申请

1. 用户进入招新页，系统只展示已发布或开放中的招新批次。
2. 用户选择批次，进入申请页。
3. 页面加载社团信息、批次时间、部门和后台配置的动态字段。
4. 用户至少选择第一志愿部门，并填写姓名、联系方式和所有必填字段。
5. 可选第二志愿不得与第一志愿相同。
6. 提交前进行字段校验，并展示个人信息使用提示。
7. 后端校验批次状态、时间窗口、重复申请和字段规则。
8. 登录用户提交成功后进入申请进度；游客提交成功后返回招新页并展示查询提示。

首版建议：允许后台控制每个招新批次是否支持游客申请。若游客提交后没有可靠的查询凭证，应强制登录，避免用户无法查看进度。

### 7.4 申请进度

1. 登录用户进入“我的申请”。
2. 页面读取当前用户的申请和相关面试信息。
3. 按最新更新时间倒序展示。
4. 每张申请卡展示社团、批次、志愿、提交时间、当前状态和面试安排。
5. 空状态引导用户前往招新页；未登录状态引导登录。

建议统一状态：

| 状态值 | 用户文案 | 说明 |
| --- | --- | --- |
| `draft` | 待提交 | 已创建但未正式进入审核 |
| `submitted` | 已提交 | 等待初步处理 |
| `screening` | 筛选中 | 正在进行材料筛选 |
| `interview` | 待面试 | 已安排或等待面试 |
| `approved` | 已录取 | 申请通过 |
| `rejected` | 未通过 | 申请未通过 |
| `withdrawn` | 已撤回 | 用户主动撤回 |

后端现有状态若与上表不同，应建立统一映射，不允许页面直接展示英文状态值。

### 7.5 扫码签到

1. 登录用户进入签到页或直接扫描活动现场二维码。
2. 客户端读取二维码内容并提取签到 token。
3. 客户端调用 `POST /site/check-ins/scan`。
4. 后端校验用户、签到场次、token 有效期、签到时间窗和重复签到。
5. 成功后展示活动名称、签到时间和结果；失败时展示明确原因。

失败原因至少覆盖：未登录、二维码无效、二维码过期、签到未开始、签到已结束、重复签到、无参与资格和网络异常。

### 7.6 动态表单

1. 用户通过后台分享的小程序路径进入表单页。
2. 页面依据 formId 加载表单标题、说明、开放时间、状态和字段定义。
3. 根据字段类型渲染输入控件，并对必填、格式、长度和选项进行校验。
4. 用户确认后提交，按钮进入加载状态并禁止重复点击。
5. 成功后展示后台配置的成功文案，默认返回首页。

P0 支持字段：单行文本、多行文本、单选、下拉、日期、手机号、邮箱、数字。多选、图片和文件上传列为 P1。

## 8. 页面功能规格

### 8.1 首页

展示模块：

- 社团 Banner：名称、简介、封面及“加入我们”入口。
- 关键数据：成员、活动、项目或后台配置指标。
- 关注方向：技术或社团方向标签。
- 近期活动：横向卡片，点击进入活动详情。
- 成员/指导老师展示。
- 荣誉成果展示。

交互与验收：

- 首次进入自动加载，支持下拉刷新。
- 任一非关键模块为空时不影响其他模块展示。
- 整页失败时显示错误态和重试按钮，不以空白页面代替错误。
- “加入我们”进入招新 Tab；“查看活动”滚动到活动区或进入活动 Tab。
- 图片加载失败时显示默认占位图。

### 8.2 活动列表

功能要求：

- 展示标题、封面、时间、地点、状态、摘要和标签。
- 支持全部、报名中、即将开始、已结束等状态筛选。
- 支持下拉刷新和触底加载；分页结果去重。
- 点击卡片进入详情。
- 空数据和接口失败分别展示不同状态。

### 8.3 活动详情

功能要求：

- 展示封面、标题、状态、时间范围、地点和富文本/Markdown 介绍。
- 展示报名开始、截止、名额和当前报名状态。
- 报名按钮遵守第 7.2 节状态规则。
- 报名请求防重复提交；成功后必须重新拉取报名状态或本地可靠更新。
- 支持微信好友分享，分享标题、封面和页面路径来自活动数据。

### 8.4 招新列表

功能要求：

- 展示开放中的批次，并可展示未开始和已结束批次作为信息预告。
- 卡片展示社团、批次名、申请时间、状态和简介。
- 开放批次可进入申请，其他状态显示原因。
- 支持刷新、加载、空状态和错误重试。

### 8.5 招新申请

功能要求：

- 动态渲染部门志愿和后台配置字段。
- 姓名、联系方式必须存在；若动态 schema 未定义，客户端补充基础字段。
- 第一志愿必填；第二志愿可选且不得重复。
- 必填项去除首尾空格后校验；手机号、邮箱等字段按类型校验。
- 提交按钮仅在批次有效期内可用。
- 提交期间禁止返回造成重复提交，失败后保留已填内容。
- 提交成功后清理草稿；失败时不得清空表单。

### 8.6 消息

功能要求：

- 未登录时显示登录引导。
- 登录后展示标题、摘要、发送时间、已读状态和消息类型。
- 点击未读消息时先标记已读，再按业务目标跳转；跳转失败仍可查看正文。
- 支持全部已读；操作成功后同步更新 Tab 角标和个人中心角标。
- 下拉刷新时重新获取消息列表和未读数。
- 无消息与加载失败使用不同提示。

### 8.7 我的

功能要求：

- 游客显示登录入口；登录用户显示头像、姓名和身份摘要。
- 菜单包括我的申请、扫码签到、招新报名、消息通知、近期活动和微信绑定。
- 角标展示申请数量和未读消息数。
- 微信已绑定时显示绑定状态；P0 不提供直接解绑，解绑由安全设置或客服流程处理。
- 退出登录必须二次确认。
- 增加用户协议、隐私政策、关于我们和版本号入口。

### 8.8 登录

功能要求：

- 支持账号密码登录和微信快捷登录。
- 密码默认隐藏，可切换显示状态。
- 登录按钮防重复点击；失败时展示后端返回的可理解文案。
- 登录成功后保存 Token、用户、角色和权限。
- 用户资料缺失时提示补充资料，但不得造成无法返回的死循环。
- 若后台关闭注册，隐藏或禁用注册入口。

### 8.9 申请进度、签到与动态表单

分别遵循第 7.4、7.5、7.6 节流程，并统一具备：加载状态、空状态、错误重试、登录拦截、防重复提交和安全区适配。

## 9. 业务规则

### 9.1 时间

- 后端统一存储带时区或 UTC 时间，客户端按用户所在时区显示。
- 所有报名、申请和签到时间窗以后端时间为最终判断依据，前端判断仅用于按钮展示。
- 时间范围采用“开始时间 — 结束时间”，同日时可省略重复日期。

### 9.2 重复提交与幂等

- 活动报名、招新申请、动态表单和签到均需前端锁定按钮。
- 后端应通过唯一约束或幂等键防止网络重试产生重复记录。
- 重复报名或签到应返回业务结果，例如“你已报名”，而非通用服务器错误。

### 9.3 状态与文案

- 状态值由后端保持稳定，客户端使用集中映射转换为中文文案与颜色。
- 未识别状态使用中性样式和“处理中”，同时记录监控事件。
- 不在多个页面各自维护不同的状态翻译。

### 9.4 内容安全

- Markdown/富文本必须过滤脚本、事件属性、危险 URL 和不受信任 iframe。
- 用户输入在展示端进行转义，后端同时执行长度和内容校验。
- 图片域名必须加入微信业务域名白名单，并为失败场景提供占位图。

- 活动报名名额、报名时间窗口及重复报名错误码。
- 消息“全部已读”批量接口；若暂不提供，客户端只能逐条调用，不适合大量消息。
- 消息详情或消息正文、目标业务类型与目标 ID。
- 游客招新申请的查询机制，或改为强制登录。
- 所有写接口的幂等策略和业务错误码。
- 动态表单字段类型、校验规则、成功页配置的统一 schema。

## 11. 通用交互规范

### 11.1 页面状态

每个数据页面必须实现四态：

1. 首次加载：骨架屏或明确的加载提示。
2. 空数据：说明当前没有内容，并给出可执行的下一步。
3. 加载失败：展示简短原因和重试按钮。
4. 正常内容：刷新时保留现有内容，避免整页闪烁。

### 11.2 错误提示

- 表单校验使用字段附近提示；全局业务错误使用 Toast 或结果页。
- 网络超时提示“网络连接失败，请重试”。
- 无权限提示“当前账号暂无权限”。
- 登录过期提示“登录已过期，请重新登录”，确认后进入登录页。
- 后端内部错误不得直接暴露堆栈、SQL 或第三方接口原文。

### 11.3 返回与导航

- Tab 间使用 `reLaunch` 或自定义 Tab 切换，详情页使用 `navigateTo`。
- 分享直达页面必须支持无历史栈场景，返回失败时回到对应 Tab 或首页。
- 登录、提交成功和外部分享场景必须定义明确的回退页面。

## 12. 非功能需求

### 12.1 性能

- 首包应控制在微信小程序限制内，非首屏页面优先分包。
- 首页可交互时间建议 ≤ 2 秒（常规 4G/Wi-Fi，接口正常）。
- 列表首屏建议单次 10—20 条，图片懒加载并使用适当尺寸。
- 避免首页一次加载无上限的成员、荣誉和活动数据。

### 12.2 安全与隐私

- AppSecret、数据库密码、服务端密钥不得出现在小程序包、前端环境变量或公开文档中。
- 生产环境必须启用微信合法域名校验和 HTTPS，禁止以 `urlCheck: false` 作为发布配置。
- 日志不得记录密码、Token、微信 code、openid、手机号全文或完整表单内容。
- 上线前提供隐私政策和用户协议，明确收集的账号、联系方式、申请和签到数据用途。
- 仅在用户触发扫码时申请相机权限；拒绝后说明用途并允许再次操作。
- 遵守最小权限原则，不申请业务未使用的微信能力。

### 12.3 兼容性与可访问性

- 覆盖当前微信支持的主流 iOS、Android 版本和常见屏幕尺寸。
- 适配状态栏、刘海屏、底部安全区和系统字体放大。
- 文字与背景保持足够对比度，关键信息不能只依赖颜色表达。
- 可点击区域建议不小于 44×44 CSS 像素等效尺寸。

### 12.4 可观测性

- 记录页面加载成功/失败、登录结果、活动报名、招新提交、表单提交和签到结果。
- 埋点不采集自由文本、联系方式、Token、openid 或完整二维码内容。
- 前端异常上报包含版本、页面、错误类型和网络状态，不包含敏感业务数据。

## 13. 验收标准

### 13.1 P0 功能验收

- 游客可正常浏览首页、活动和招新公开内容。
- 账号密码登录和微信登录均可建立有效会话；退出后受限数据被清理。
- 活动在不同状态下显示正确按钮，重复报名不会生成重复记录。
- 招新申请可按动态 schema 渲染、校验并提交，失败后数据不丢失。
- 登录用户能查看自己的申请和面试进度，不能访问他人记录。
- 消息未读数在消息页、Tab 和个人中心保持一致。
- 有效二维码可签到，过期、重复和无权限场景返回清晰结果。
- 后台分享的招新和表单路径可从微信冷启动直达目标页。
- 所有核心页面具备加载、空数据、错误和重试状态。

### 13.2 上线前检查

- `pnpm run typecheck` 通过。
- 小程序开发版、体验版完成真机验证。
- 微信请求、上传、下载及图片域名全部配置完成。
- 隐私协议、用户协议、类目、图标、截图和版本说明齐全。
- 生产环境关闭调试配置并完成密钥扫描。
- 后端接口完成鉴权、限流、幂等和敏感日志检查。
- 至少覆盖登录、活动报名、招新申请、消息、动态表单和签到的回归用例。

## 14. 建议开发拆分

### 阶段一：工程与公共能力

- 环境配置、请求封装、登录态、401 拦截和返回原页面。
- 公共导航、Tabbar、状态组件、图片占位和状态映射。
- 生产/测试环境隔离、埋点和异常上报。

### 阶段二：公开浏览链路

- 首页。
- 活动列表与详情。
- 招新列表与招新详情。
- 分享路径和冷启动参数处理。

### 阶段三：登录与提交链路

- 账号密码登录、微信登录和绑定。
- 活动报名。
- 招新申请与进度。
- 动态表单。

### 阶段四：成员高频能力

- 消息与未读同步。
- 扫码签到。
- 个人中心和退出登录。

### 阶段五：测试与上线

- 接口联调、状态矩阵、异常网络和真机兼容测试。
- 隐私合规、合法域名、性能优化和体验版验收。
- 灰度发布、监控和回滚预案。

## 15. 开发前待确认事项

以下事项会直接影响实现，应在正式排期前定稿：

1. 小程序首版是否只服务一个默认社团，还是需要用户选择学校/社团。
2. 游客是否允许提交招新和动态表单；若允许，如何查询申请结果。
3. 微信快捷登录遇到未绑定 openid 时，是自动创建账号还是要求绑定已有账号。
4. 活动是否有名额、候补、取消报名和审核机制。
5. 消息点击后的目标页面类型及参数协议。
6. 动态表单 P0 的最终字段类型和数据校验 schema。
7. 是否在首版接入微信订阅消息，以及需要申请的消息模板。
8. 现有后端各业务状态值、错误码和分页响应是否已经统一。

## 16. 现有代码基线说明

仓库当前已存在首页、活动、招新、消息、个人中心、登录、申请进度、动态表单和签到页面，以及相关 API 封装。开发应以现有页面为起点进行补齐和重构，不建议重新创建一套平行小程序。

当前实现已经覆盖主要页面外壳，但上线前仍应重点补齐：完整登录回跳、游客申请策略、活动报名状态回显、消息全部已读接口、个人协议入口、统一错误态、分享能力、生产安全配置及系统化测试。

## 17. V1 默认决策与首批开发任务

为避免等待全部产品问题定稿后才能开工，若无新的业务决定，V1 暂按以下口径开发：

| 决策项 | V1 默认口径 |
| --- | --- |
| 社团范围 | 首版展示一个后台默认社团，接口保留 `clubId` 扩展能力 |
| 公开访问 | 首页、活动、招新和开放表单允许游客浏览 |
| 个人操作 | 活动报名、进度、消息、签到必须登录 |
| 招新提交 | 延续现有实现，后台开放的招新允许游客提交；页面明确提示游客无法在线查看进度 |
| 微信首次登录 | openid 未绑定时，引导用户登录已有账号后完成绑定，不自动创建资料不完整的账号 |
| 活动机制 | 首版不做候补和取消报名；报名名额及重复报名由后端校验 |
| 动态表单 | P0 支持文本、多行文本、单选、下拉、日期、手机号、邮箱、数字 |
| 消息触达 | 首版先完成站内消息，不阻塞于微信订阅消息模板申请 |
| 多语言 | 首版使用简体中文，文案集中管理，保留后续国际化能力 |

第一批任务按以下顺序执行：

1. 完善请求层：统一响应解包、超时、业务错误、401 处理和登录后回跳。
2. 建立公共状态：登录用户、未读数、当前社团和页面返回地址。
3. 建立公共组件：页面加载、空状态、错误重试、登录拦截和提交结果反馈。
4. 打通首页、活动列表和活动详情的公开浏览链路。
5. 打通微信登录、账号绑定和活动报名，并补报名状态回显。
6. 打通招新申请、申请进度和动态表单，补齐重复提交与失败保留。
7. 完成消息未读同步、扫码签到和个人中心安全退出。
8. 完成分享直达、隐私协议、生产环境配置、真机测试和上线检查。

首批开发完成的判定条件：用户从微信冷启动进入小程序后，能连续完成“浏览活动 → 登录 → 报名 → 查看消息”以及“打开招新分享 → 填写 → 提交 → 登录后查看进度”两条主链路，中间不存在白屏、无法返回、重复记录或数据丢失。
